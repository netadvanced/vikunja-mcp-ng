# Vikunja API Coverage & Correctness Audit

Originally audited against the live Vikunja OpenAPI spec (`GET /api/v1/docs.json` on try.vikunja.io, spec version `v2.3.0-1019-g95b7e673`), the `node-vikunja` client library (v0.4.0) this project wrapped at the time, and this repo's `src/tools/` at commit `29f4f47` (branch `main`). Generated by an 11-agent audit, one per API domain, each reading the spec, the node-vikunja source, and the corresponding MCP tool code directly (not from memory). This paragraph is kept as historical provenance for the document's origin ‚Äî see the "Updated as of" line below for what the document currently reflects.

> `node-vikunja` has since been **fully removed** (PR #73, Wave D): all HTTP now goes through `vikunjaRestRequest` / `vikunjaRestMultipartRequest` against types generated from the vendored OpenAPI spec (`docs/vikunja-openapi.json`), and the dependency is gone from `package.json` (`grep -rn node-vikunja src/ package.json package-lock.json` returns zero hits). Every row and note below has been re-verified directly against current `main` as part of this pass; references to `node-vikunja` methods/types/bugs have been removed except where a source comment itself frames a past design decision historically ‚Äî those are labelled "the legacy client", matching the convention current code comments (e.g. `src/utils/label-bulk.ts`, `src/tools/projects/duplicate.ts`, `src/tools/projects/views.ts`) already use for the same purpose.

**Updated as of:** `main` @ `ce81bd7`, 2026-07-18 ‚Äî full row-by-row re-verification against current `src/tools/**` and `src/utils/vikunja-rest.ts`, done after the node-vikunja removal and Waves A‚ÄìF (see `docs/ROADMAP.md` ¬ß2/¬ß6 for wave history). ## Composite / MCP-only Features

These MCP tools don't map 1:1 to a single Vikunja endpoint ‚Äî they compose several documented calls into one higher-level operation. Listed separately since "not implemented" / "implemented" doesn't apply cleanly.

| Feature | Files | Notes |
|---|---|---|
| vikunja_tasks (`bulk-create`) | `src/tools/tasks/bulk-operations-simplified.ts` | Composes `PUT /projects/{id}/tasks` per task + `POST /tasks/{id}/labels/bulk` (via `setTaskLabels`) + `PUT /tasks/{id}/assignees` per assignee. All underlying calls verified correct against the OpenAPI schemas. Creates run **sequentially** (`processors.create` at `maxConcurrency: 1`, fixed by #116, reported by @angusmaul): on SQLite-backed Vikunja, concurrent creates 500 with an unclassifiable "database is locked" body, the 5xx retry re-storms the same contended pool, and the resulting failure burst trips the shared circuit breaker ‚Äî an outage, not just a partial failure. This **supersedes PR #97's "safe-as-is" verdict** for this call site, which had assumed bounded-at-8 plus retry was sufficient; live SQLite evidence showed it wasn't. |
| vikunja_tasks (`bulk-update`) | `src/tools/tasks/bulk-operations-simplified.ts`, `src/tools/tasks/bulk/BulkOperationValidator.ts` | Scalar fields (`done`, `priority`, `due_date`, `start_date`, `end_date`, `project_id`, `repeat_after`, `repeat_mode`) go through native `POST /tasks/bulk` (`models.BulkTask`) as one request (PR #89), snapshotting and restoring assignees around it ‚Äî **verified mechanism**: `updateTaskAssignees` runs before the `fields` gate and reconciles to `values.assignees`, which is `nil` for scalar-only bulk requests, triggering a full delete (`task_assignees.go`'s full-delete branch) for every task in `task_ids`, regardless of `fields` ‚Äî and falling back to per-task `GET`+merge+`POST` if the server doesn't honor `fields`/`values`. The restore step itself is ONE `POST /tasks/{taskID}/assignees/bulk` (`models.BulkAssignees`, replace semantics) call per task carrying that task's complete snapshotted assignee list, not a per-user `PUT` loop ‚Äî safe here because a complete snapshot is being restored (see the `/tasks/{taskID}/assignees/bulk` row above for why replace semantics stay a footgun everywhere else). `assignees`/`labels` as the primary bulk-update `field` always use the per-task `PUT`/`DELETE /tasks/{id}/assignees` or `setTaskLabels` path (the bulk endpoint doesn't support them). The response's success/count/failure fields are derived from what the server actually returned, never the requested `taskIds`, and assignee-restore failures are surfaced (`assigneeRestoreFailures`) rather than only logged. `BulkOperationValidator` rejects any `field` outside this allowlist (e.g. a camelCase `dueDate`) up front, listing the valid names. `field='repeat_mode'` used to validate against a hardcoded list that didn't match the actual value-conversion table (a HIGH-severity bug in the original audit ‚Äî `'day'`/`'week'`/`'year'` passed validation but weren't convertible, while the real enum values `'default'`/`'from_current'` were rejected outright). `validModes` is now derived directly from `REPEAT_MODE_MAP` (`src/tools/tasks/constants.ts`), so the two can never drift again ‚Äî fixed. |
| vikunja_tasks (`bulk-delete`) | `src/tools/tasks/bulk-operations-simplified.ts` | Composes `GET /tasks/{id}` (snapshot) then `DELETE /tasks/{id}` per task ID. Correct; partial failure is reported with successful/failed ID lists. |
| vikunja_tasks (`create-subtask`, `list-subtasks`) | `src/tools/tasks/subtasks.ts` | `create-subtask` resolves the parent task to inherit its project, creates the new task in that project, optionally attaches labels/assignees and places it in a Kanban bucket, creates the parent‚Üíchild `subtask` relation via `PUT /tasks/{taskID}/relations`, then re-reads the parent to verify the relation landed ‚Äî built on `CompositeOperation` (best-effort by default; `atomic: true` opts into rollback). `list-subtasks` is a read composite: one `GET /tasks/{id}` call filtered to the `"subtask"` slice of `related_tasks`. Not a documented single endpoint ‚Äî Vikunja has no first-class "subtask" resource. |
| vikunja_projects (`share-with-user`, `share-with-team`, `list-members`) | `src/tools/projects/sharing-access.ts` | Composites over the direct user/team sharing primitives above: `share-with-user`/`share-with-team` resolve a username/team name to an id (searching if needed), then add-or-update the permission in one call (ensure-semantics); `list-members` combines `GET /projects/{id}/users` and `GET /projects/{id}/teams` into one summarized response. |

`vikunja_export`, `vikunja_batch_import`, and `vikunja_templates` are also composite (built from task/project/label CRUD calls, not a dedicated endpoint) ‚Äî see the Issues table above for open findings in how they compose those calls (`EntityResolver.ts` batch-import user search, `export.ts` recursive project refetch, `templates.ts` session-scoped persistence).
