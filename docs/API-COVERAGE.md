# Vikunja API Coverage & Correctness Audit

Originally audited against the live Vikunja OpenAPI spec (`GET /api/v1/docs.json` on try.vikunja.io, spec version `v2.3.0-1019-g95b7e673`), the `node-vikunja` client library (v0.4.0) this project wrapped at the time, and this repo's `src/tools/` at commit `29f4f47` (branch `main`). Generated by an 11-agent audit, one per API domain, each reading the spec, the node-vikunja source, and the corresponding MCP tool code directly (not from memory). This paragraph is kept as historical provenance for the document's origin — see the "Updated as of" line below for what the document currently reflects.

> `node-vikunja` has since been **fully removed** (PR #73, Wave D): all HTTP now goes through `vikunjaRestRequest` / `vikunjaRestMultipartRequest` against types generated from the vendored OpenAPI spec (`docs/vikunja-openapi.json`), and the dependency is gone from `package.json` (`grep -rn node-vikunja src/ package.json package-lock.json` returns zero hits). Every row and note below has been re-verified directly against current `main` as part of this pass; references to `node-vikunja` methods/types/bugs have been removed except where a source comment itself frames a past design decision historically — those are labelled "the legacy client", matching the convention current code comments (e.g. `src/utils/label-bulk.ts`, `src/tools/projects/duplicate.ts`, `src/tools/projects/views.ts`) already use for the same purpose.

**Updated as of:** `main` @ `ce81bd7`, 2026-07-18 — full row-by-row re-verification against current `src/tools/**` and `src/utils/vikunja-rest.ts`, done after the node-vikunja removal and Waves A–F (see `docs/ROADMAP.md` §2/§6 for wave history). The incremental "Updated 2026-07-1x (Wave D, ...)" notes that used to accumulate in this header have been superseded by this full pass and removed; that history is preserved in git/PR history (and issue #28), not duplicated here.

**Maintenance rule:** a row's status flips in the same PR that changes the underlying coverage (implementing, fixing, or removing a capability) — never as a separate later "audit" pass. The summary table below is always **recounted** directly from the row status markers (e.g. `grep -cE '^\| (GET|PUT|POST|DELETE|PATCH) \|.*✅' docs/API-COVERAGE.md` per status), never hand-adjusted independently of the rows.

## Summary

Vikunja documents **169** distinct API operations (method+path). Of those:

| Status | Count | % |
|---|---|---|
| ✅ Implemented | 103 | 61% |
| ⚠️ Implemented (bug) | 1 | 1% |
| 🟡 Partial | 1 | 1% |
| ❌ Not implemented | 64 | 38% |
| **Total** | **169** | 100% |

- **✅ Implemented** — implementation matches the documented endpoint (method, path, request/response fields all checked).
- **⚠️ Implemented (bug)** — the tool exists and calls something, but the audit found a concrete divergence from the documented contract (wrong field name, dropped required param, wrong response shape assumption, etc.). See the Issues table below.
- **🟡 Partial** — works for the common case but silently drops documented capability (e.g. missing pagination/search params), or is an honest, documented, by-design partial (e.g. no binary content channel).
- **❌ Not implemented** — no MCP tool calls this endpoint at all. Some are intentional (credential ceremonies, genuinely-binary blob endpoints, migration importers, destructive test/account endpoints) rather than gaps; others are approved-to-build but not yet coded. See notes per row and the re-triage below.

> **C2 endpoint-tail re-triage (2026-07-18):** all 64 `❌` rows were re-examined under the direct-REST architecture (`node-vikunja` removed). Several 2026-07-17 "won't-implement" verdicts rested on limitations that no longer apply (a broken client-library call, a blanket "binary/blob" rule that mislabeled JSON endpoints). New split: **20 IMPLEMENT (approved-to-build) / 36 PARKED / 8 NEVER** — full per-op verdicts, spec evidence, and the grouped candidate backlog (G1–G7) live in **[docs/ENDPOINT-TAIL-RETRIAGE.md](ENDPOINT-TAIL-RETRIAGE.md)**. This changed **no counts**: all 64 stay `❌` until each IMPLEMENT row's code actually lands (per the maintenance rule above). The Notes on the 20 IMPLEMENT rows below are updated to carry the new verdict; PARKED/NEVER rows keep their `❌` with reasons re-justified in the re-triage doc.

## Correctness Issues

Issues found in code paths that ARE implemented (as opposed to simply missing) — bugs, not coverage gaps. This table only lists issues **still true today**; the original audit found 36, and every `node-vikunja`-only issue (moot now that the dependency is gone) plus every issue fixed by Wave A–F code changes has been removed — that history lives in git/PR history and issue #28, not here. Every ⚠️ row in the endpoint tables below has a corresponding row in this table.

| Severity | Domain | File | Description |
|---|---|---|---|
| MEDIUM | Migration importers, testing endpoints, and MCP-only composite features (export, batch-import, templates) | `src/services/EntityResolver.ts` | `fetchUsers()` calls `GET /users` with no search term (`s`) to build the username→ID map used for batch-import assignee resolution. Per the OpenAPI spec, `GET /users` is a *search* endpoint ("Search for a user by its username, name or full email") — it is not documented as "list all users I can see". Calling it with an empty/absent search criteria is very likely to return zero or near-zero results on a real Vikunja instance, silently breaking assignee-by-username resolution for essentially every batch import. The code's own comment acknowledges this as a known, deliberately out-of-scope issue: the only failure handling path treats a fetch failure as an auth problem (`userFetchFailedDueToAuth`), so a valid-but-empty search result is misclassified as "succeeded with 0 users" rather than "this call needs a search string per user", and every assignee is reported "not found" with no explanation pointing at the real cause. |
| LOW | User account & settings | `src/tools/users.ts` | The `search` subcommand's `page`/`perPage` parameters are still forwarded to `GET /users` as `page`/`per_page` query params, but the OpenAPI spec documents only a single `s` query parameter for that endpoint — pagination may not be honored by the server for user search. |
| LOW | Projects — CRUD, backgrounds, views, duplication | `src/tools/projects/hierarchy.ts` | `getProjectTree()`, `getProjectBreadcrumb()`, `getProjectChildren()`, and `moveProject()` all still fetch "all projects" via a single `GET /projects?per_page=1000` call (`fetchAllProjects()` in `crud.ts`) to walk parent/child relationships client-side. This is a performance/scale concern (the hardcoded 1000-item cap silently truncates hierarchy correctness for instances with more projects) rather than a spec-correctness bug, but it still affects the accuracy of tree/breadcrumb/move-cycle validation for large instances. |
| LOW | Project sharing — users, teams, share links | `src/tools/projects/sharing.ts` | `listProjectShares()` still never exposes the OpenAPI `s` (search-by-hash) query parameter for `GET /projects/{project}/shares`, so callers cannot filter/search link shares server-side even though the API supports it. |
| LOW | Saved filters, webhooks, subscriptions, notifications, reactions | `src/tools/webhooks.ts` | The `list` subcommand still never forwards the spec's optional `page`/`per_page` query parameters for `GET /projects/{id}/webhooks`, so callers cannot paginate through large webhook lists. |
| LOW | Migration importers, testing endpoints, and MCP-only composite features (export, batch-import, templates) | `src/tools/export.ts` | `exportProjectRecursive()` still calls `GET /projects` (fetching the caller's entire accessible project list) inside the recursion for every child project when `includeChildren` is true, purely to client-side filter by `parent_project_id`. For a deep/wide project tree this is an O(depth) full-catalog refetch; functionally correct but wasteful, as the file's own top-of-file memory-usage warning already flags generically. |
| LOW | Migration importers, testing endpoints, and MCP-only composite features (export, batch-import, templates) | `src/tools/templates.ts`, `src/storage/templateFileStore.ts` | Templates are still persisted through the session-scoped, in-memory `SimpleFilterStorage` (via `storage.create({name, filter: JSON.stringify(...), isGlobal: true})`), repurposing the saved-filter storage layer to hold template blobs. Unlike the original finding, this is no longer *unconditionally* durability-losing: an opt-in JSON file store (`templates.persistPath` config key / `VIKUNJA_MCP_TEMPLATES_FILE` env var, `src/storage/templateFileStore.ts`) now hydrates and persists that in-memory storage across restarts when configured. It remains **session-only by default** — without `persistPath` set, templates still silently vanish on restart with no user-facing warning in the tool description or responses. |

If none of the above survive a future re-verification, replace this table with a one-line pointer to issue #28's post-removal capability queue rather than leaving a stale row in place.

## Composite / MCP-only Features

These MCP tools don't map 1:1 to a single Vikunja endpoint — they compose several documented calls into one higher-level operation. Listed separately since "not implemented" / "implemented" doesn't apply cleanly.

| Feature | Files | Notes |
|---|---|---|
| vikunja_tasks (`bulk-create`) | `src/tools/tasks/bulk-operations-simplified.ts` | Composes `PUT /projects/{id}/tasks` per task + `POST /tasks/{id}/labels/bulk` (via `setTaskLabels`) + `PUT /tasks/{id}/assignees` per assignee. All underlying calls verified correct against the OpenAPI schemas. |
| vikunja_tasks (`bulk-update`) | `src/tools/tasks/bulk-operations-simplified.ts`, `src/tools/tasks/bulk/BulkOperationValidator.ts` | Scalar fields (`done`, `priority`, `due_date`, `start_date`, `end_date`, `project_id`, `repeat_after`, `repeat_mode`) go through native `POST /tasks/bulk` (`models.BulkTask`) as one request (PR #89), snapshotting and restoring assignees around it (the endpoint clears them server-side) and falling back to per-task `GET`+merge+`POST` if the server doesn't honor `fields`/`values`. `assignees`/`labels` always use the per-task `PUT`/`DELETE /tasks/{id}/assignees` or `setTaskLabels` path (the bulk endpoint doesn't support them). The response's success/count/failure fields are derived from what the server actually returned, never the requested `taskIds`, and assignee-restore failures are surfaced (`assigneeRestoreFailures`) rather than only logged. `BulkOperationValidator` rejects any `field` outside this allowlist (e.g. a camelCase `dueDate`) up front, listing the valid names. `field='repeat_mode'` used to validate against a hardcoded list that didn't match the actual value-conversion table (a HIGH-severity bug in the original audit — `'day'`/`'week'`/`'year'` passed validation but weren't convertible, while the real enum values `'default'`/`'from_current'` were rejected outright). `validModes` is now derived directly from `REPEAT_MODE_MAP` (`src/tools/tasks/constants.ts`), so the two can never drift again — fixed. |
| vikunja_tasks (`bulk-delete`) | `src/tools/tasks/bulk-operations-simplified.ts` | Composes `GET /tasks/{id}` (snapshot) then `DELETE /tasks/{id}` per task ID. Correct; partial failure is reported with successful/failed ID lists. |
| vikunja_tasks (`create-subtask`, `list-subtasks`) | `src/tools/tasks/subtasks.ts` | `create-subtask` resolves the parent task to inherit its project, creates the new task in that project, optionally attaches labels/assignees and places it in a Kanban bucket, creates the parent→child `subtask` relation via `PUT /tasks/{taskID}/relations`, then re-reads the parent to verify the relation landed — built on `CompositeOperation` (best-effort by default; `atomic: true` opts into rollback). `list-subtasks` is a read composite: one `GET /tasks/{id}` call filtered to the `"subtask"` slice of `related_tasks`. Not a documented single endpoint — Vikunja has no first-class "subtask" resource. |
| vikunja_projects (`share-with-user`, `share-with-team`, `list-members`) | `src/tools/projects/sharing-access.ts` | Composites over the direct user/team sharing primitives above: `share-with-user`/`share-with-team` resolve a username/team name to an id (searching if needed), then add-or-update the permission in one call (ensure-semantics); `list-members` combines `GET /projects/{id}/users` and `GET /projects/{id}/teams` into one summarized response. |

`vikunja_export`, `vikunja_batch_import`, and `vikunja_templates` are also composite (built from task/project/label CRUD calls, not a dedicated endpoint) — see the Issues table above for open findings in how they compose those calls (`EntityResolver.ts` batch-import user search, `export.ts` recursive project refetch, `templates.ts` session-scoped persistence).
